# Decision Table Builder(決策表建構器)

## 目的

將業務規則系統化為決策表:給定一組條件(Criteria),應產生什麼預期結果(Action)。決策表是驗收條件(AC)與具體測試案例之間的橋樑;目標是識別**有價值、高風險的組合**,而非窮舉所有排列。

## 觸發時機

**使用**:條件組合邏輯(折扣、定價、權限、資格)、邊界值測試(「至少 X 件」「超過 Y 元」)、多物件交互(多品項、多角色、多優惠疊加)。

**不使用**:效能/壓力/安全測試、狀態機與併發競態、無條件分支的順序流程。

## 規則

所有產出必須同時滿足以下規則:

1. **Criteria 用具體變數與具體值**:每個 Criteria 使用具體名稱(如「啤酒數量」)與具體值域(如 `0, 5, 6, 7, 12`),值域聚焦邊界值。二元條件也要寫出實際狀態名稱(如「折扣碼狀態: 有效 / 過期」),而非 Y/N。
2. **Action 寫出可驗證的精確結果**:有計算就寫公式(`總優惠 = 優惠組數 × -26 元`);有狀態變更就寫明確狀態名(`訂單狀態: 成立`);有上限就寫數值(`最大限購: 6 件`)。公式與單位寫在欄名中,表格儲存格只填純數值或狀態名(寫 `80`,不寫 `80 元`)。
3. **Rule 依固定優先序挑選**:(a) 邊界條件(剛好低於/等於/超過閾值)→ (b) 零值/空值/無效值 → (c) Happy Path → (d) 交互組合。挑選業務意圖明確且彼此不同的組合即可。
4. **決策表前列出「遵守規則」清單**:以列點寫出本表涵蓋的業務規則(如「啤酒每滿 6 罐折 26 元」),所有情境共同遵守;每條規則至少被一個情境驗證。
5. **一張表最多 8 個 Rule**:超過時依情境群組拆成多張子決策表,每張表有自己的名稱。
6. **每個 Rule 是一列,以語意化情境名稱命名**:Criteria 與 Action 合併為單一表格,每個 Rule 佔一列(row),列名為 2–8 字的具體情境名稱(如「差1元未達」「恰好滿額」「空購物車」「VIP未滿額」),同表內名稱不重複;列由上而下依規則 3 的優先序排列;欄由左至右為:情境 → 條件欄(依變數在需求描述中出現的順序)→ 結果欄。
7. **明確標記假設與不確定處**:輸出末尾列出你所做的假設、不確定的業務邏輯、可能遺漏的邊緣情況,請領域專家 Review。數值計算一律請用戶驗證公式與預期值。

## 固定輸出格式

每張決策表一律使用以下三段式格式,段落順序與標題文字固定。輸出以 `## 決策表:[名稱]` 為最上層標題(前面不加其他標題);情境為列(由上而下依規則 3 優先序)、條件與結果為欄(單一合併表格),情境很多時表格向下延伸、不影響閱讀寬度:

```markdown
## 決策表:[功能/規則名稱]

### 遵守規則
- [業務規則 1]
- [業務規則 2]

### 情境與預期結果(Criteria → Action)
| 情境        | [變數名稱1] | [變數名稱2] | [結果名稱1(含公式/狀態名)] | ... |
|-------------|-------------|-------------|------------------------------|-----|
| [情境名稱1] | 值          | 值          | 值                           |     |
| [情境名稱2] | 值          | 值          | 值                           |     |

### 待確認事項
- [假設 / 不確定的邏輯 / 可能遺漏的情境]
```

## 工作流程

1. **釐清範圍**:從用戶描述提取核心業務邏輯、影響結果的輸入變數、預期結果、已知邊界值、操作介面(API/Web/App)。描述簡短時,先合理推斷並列出假設請用戶確認。
2. **定義 Criteria**:依規則 1 列出變數與值域。
3. **定義 Action**:依規則 2 寫出精確結果。
4. **挑選 Rules**:條件數 ≥ 2 時,先執行組合展開 script(見「輔助工具」)列出值域的全部組合,再依規則 3 的優先序從中挑選;整理「遵守規則」清單並確認每條規則至少被一個情境驗證(規則 4)。
5. **輸出決策表**:套用「固定輸出格式」,依規則 5–7 檢查。
6. **邀請 Review**:呈現「待確認事項」,請領域專家確認後才算定案。

## 輔助工具:組合展開 script

條件數 ≥ 2 時,執行本 skill 目錄下的 `scripts/generate-decision-table.sh` 展開值域的笛卡兒積,輔助挑選情境並檢查覆蓋率:

```bash
bash scripts/generate-decision-table.sh \
  -c "訂單金額:0,500,999,1000,1500" \
  -c "會員身分:一般,VIP" \
  -r "EXCLUDE:訂單金額==0 && 會員身分==VIP"
```

- 輸出為 Markdown 表格、一列一個組合,與「情境與預期結果」表的列一一對應:從展開結果挑選有業務價值的組合(不必全部採用),為選中的列命名情境、補上結果欄。
- `-r "EXCLUDE:條件A==值 && 條件B==值"` 剪掉無業務意義的組合;`-r "OVERRIDE:條件A==值 && ... -> 條件B=值"` 強制覆蓋特定組合的值。
- 先依規則 1 把值域縮小到邊界值再展開;script 預估組合數 > 20 會發出警告,此時縮小值域或用 EXCLUDE 收斂。

## 簡單範例

輸入需求:「啤酒每 6 罐折 26 元,可累計。」

```markdown
## 決策表:啤酒滿 6 罐折扣

### 遵守規則
- 啤酒每滿 6 罐折 26 元
- 折扣可累計(12 罐折兩組)
- 未滿 6 罐不觸發任何折扣

### 情境與預期結果(Criteria → Action)
| 情境      | 啤酒數量 | 總優惠 = floor(數量/6) × -26 元 |
|-----------|----------|---------------------------------|
| 差1件未達 | 5        | 0                               |
| 恰好達標  | 6        | -26                             |
| 空購物車  | 0        | 0                               |
| 兩組累計  | 12       | -52                             |

### 待確認事項
- 假設優惠可無上限累計,請確認是否有累計上限。
```

## 產出前檢查

- [ ] 每個 Criteria 都是具體變數名 + 具體值域(無 Y/N)
- [ ] 每個 Action 都含公式、狀態名或數值上限
- [ ] Rule 涵蓋:邊界、零/空/無效、Happy Path
- [ ] 條件數 ≥ 2 時,已用 `scripts/generate-decision-table.sh` 展開組合,確認挑選的情境已涵蓋所有有價值的組合
- [ ] 有「遵守規則」列點段落,且每條規則至少被一個情境驗證
- [ ] 單表 ≤ 8 個 Rule,每列為不重複的語意化情境名稱(2–8 字);欄序為 情境 → 條件欄 → 結果欄
- [ ] 有「待確認事項」段落,列出假設與待驗證數值
