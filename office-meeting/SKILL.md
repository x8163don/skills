# office-meeting — 會議助手

## 這個 skill 在做什麼

涵蓋會議的三個階段,依使用者需求套用對應框架:

1. **會議前** — 幫忙設計議程結構,讓會議有機會控制在時間內、聚焦在該討論的事
2. **會議中** — 提供讓沉默會議動起來、讓與會者專心參與的具體做法
3. **會議後** — 把雜亂筆記/逐字稿整理成「一眼就懂、三秒抓重點」的會議紀錄

先判斷使用者現在處於哪個階段,再套用對應框架。如果使用者只是要把筆記整理成紀錄,直接跳到【會議後:會議紀錄四大結構】,不需要走完前面兩段。

---

## 會議前:30 分鐘法則 + 議程設計

**為什麼**:專注力在 20-30 分鐘後明顯下降;給的時間越長,問題反而越容易被討論得更複雜、更容易失焦變成閒聊或爭辯。

**三招讓會議回到 30 分鐘內解決重點**:

1. **會議前先用文件「預備發言」**:開會是為了討論,不是為了同步資訊。資訊事前用 Doc/Notion 列出議題與初步觀點,讓大家會前先讀,開會直接跳過背景介紹。
2. **會議三件事原則**:
   - 明確議題:這場會只討論「這幾件事」(建議列出來,3 件以內)
   - 控制時間:每個議題限定時間(例如 10 分鐘),時間到就收斂出下一步
   - 指定主持人:一定要有人負責主持與收斂,否則會議會變成放牛吃草
3. **卡關就會後處理,不要死磕**:當場決定下一步是「轉 1v1 溝通」「請某人先內部提案再回來對齊」還是「收斂成兩案讓主管拍板」——時間到就下會。

**輸出格式**:當使用者要求「幫我設計這場會議的議程」時,產出包含以下欄位的議程:
- 會議目的(一句話)
- 討論議題清單(每項標時間配額,加總不超過使用者給定的會議長度)
- 主持人/收斂負責人
- 會前需要先讀的資料(如果有)

## 會議中:讓會議「活起來」的做法

### 情境 A:大家都在滑手機/心不在焉

**心法**:不是講得不夠好,是讓人覺得「不用聽也沒差」。要從「報告進度」轉成「設計一場對話」。

三個做法:
1. **一開場點名「這場會議跟誰有關」**:「今天這場會議有兩個關鍵決策要拍板,其中一個會影響 A 部門下週行動,另一個會需要 B 跟 C 協作。」
2. **把「你要聽」變成「你必須參與」**:「我會提三個方案,每個人等下選一個你最支持的,並講原因,我們用投票決定。」
3. **不要念報告,設計互動**:用提問(「你們覺得這三個方向裡,哪一個風險最高?」)或具體案例取代照稿念。

實作建議:每 5-10 分鐘設一個小互動(舉手表態、小組回饋、票選),結尾請每個人自己唸一次自己的 Action Item 做確認。

### 情境 B:會議沉默、沒人開口

**心法**:沉默不是沒想法,是「沒安全感＋沒引導＋沒節奏感」。就算不是主持人,也可以主動打破沉默。

三種降低發言壓力的句型:
1. **補充＋問題模式**:「我補充一個觀察,剛剛提到的使用者需求 A,我這邊看到數據顯示其實也有 B 的情況出現,我想問一下這邊會不會也需要考慮?」
2. **緩衝語**:「我不確定這是不是合適的切入角度,但我想到一個可能⋯⋯」「我想先拋一個初步想法,還需要再補資料,但想先丟出來看看大家怎麼想」
3. **問別人而非發表自己**,把演講變對話:「剛剛這部分,我記得 XX 上週好像也有處理類似情境?能不能請你補充一下?」

判斷「該不該發聲」的三個提醒:這場會有我負責的內容 / 討論結果會影響我後續工作 / 我有觀察或數據能幫助判斷——符合任一項就該開口。

## 會議後:會議紀錄四大結構(一眼就懂、三秒抓重點)

**為什麼大家不看會議紀錄**:全篇像逐字稿、沒標題沒段落、任務沒有明確負責人與時程、連結論都看不到。記住:會議紀錄不是記錄過程,是幫助決策與執行的工具。

**輸出時一律套用這個結構**,把使用者提供的原始筆記/逐字稿整理進去:

```
## 會議基本資訊
會議名稱:
時間:
與會者:
主持人:
記錄人(可選):

## 重點結論摘要(最重要,放最前面)
- [結論/決策項目 1](負責人,時間)
- [結論/決策項目 2](負責人,時間)

## 討論紀要(不超過三層,只留有價值的內容)
### 議題一、[議題名稱]
- 討論重點:
- 初步結論:
- 延伸任務:(負責人,時間)

### 議題二、[議題名稱]
...

## 待辦事項
| 任務 | 負責人 | 期限 |
|---|---|---|
| | | |
```

**整理原則**:
- 「重點結論摘要」永遠放最前面,讓主管三秒掃完就知道「要幹嘛、誰負責、何時完成」
- 討論紀要不需要逐字還原對話,只留「有沒有結論、有沒有任務、有沒有人負責」,話是誰說的不重要
- 如果原始筆記沒有標出負責人或時程,明確反問使用者要補上,不要自己捏造
- 建議信件/文件標題格式:「會議主題_日期_版本」(例如「產品會議_2025-08-17_v2」),不要只寫「OO 會議紀錄」

---

## 產出時的通用原則

- 先判斷使用者要的是議程設計、會議帶領建議,還是紀錄整理,三者對應不同框架,不要混著給。
- 整理會議紀錄時,只根據使用者提供的內容整理,不要新增使用者沒提到的結論或任務。
- 若使用者要的是「怎麼提出異議/怎麼跟主管建議」這類會議中的個人話術問題,轉交給 `office-script` skill 處理;若是「怎麼跟主管報告會議重點」則搭配 `office-report`。
